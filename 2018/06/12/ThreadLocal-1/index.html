ThreadLocal的内存泄露

为什么会有内存泄露？
内存泄露是由于不再使用的对象无法回收大量堆积，在 ThreadLocal 的使用中，ThreadLocalMap 对 ThreadLocal的引用为弱引用，当下一次垃圾回收时，会将ThreadLocal 回收。但此时如果持有 Thread 对象的线程还未结束，那么 Thread 依然会持有 ThreadLocalMap， ThreadLocalMap 的值 Value 也会继续存在，但此时，Value 已经是一个 key 为 null 的值了。从上面的描述看起来好像是由于弱引用的关系导致了内存泄露，其实不然，这是引用的生命周期问题，当持有一个引用的对象生命周期长于被引用的对象，而被引用对象又没有清除时就会导致内存泄露，所以这根本不是弱引用的问题。在 ThreadLocal 中为了避免这种情况，每次在 set、get、remove 的时候都会清楚其中的 key 为 null 的情况。

根据方法名，我们可以看出哪些方法是用来清除那些 key 为 null 的情况，stale 中文意思为过期的、失效的。因此我们发现，只要每次用完后进行 set、get、remove 操作都会有效避免内存泄露。那为何使用弱引用？思考下发现，其实弱引用避免了我们去手动的删除 ThreadLocal，因为如果是强引用，那么 只要持有 Thread 对象的线程存在，那么 ThreadLocalMap 也会存在，从而强引用 ThreadLocal，当不需要某个 ThreadLocal 的时候，如果没有手动删除，那么造成 Entry 迟迟无法清空，此时如果线程中有更多的 ThreadLocal 加入进来，那么会占用数组 table 的位置，从而导致数组扩容，到一定程度后造成内存泄露。
因此为了避免该问题，请在合适的时刻调用 remove、set、get 方法
